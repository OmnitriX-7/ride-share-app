Master Backend Logic: Full-Scale Ride-Sharing System

Phase 1: Identity & Authentication (The Security Layer)

1.1 User Registration Logic

Input Validation:
The backend must utilize a validation schema (like Zod or Joi). It checks if the email is a valid string format, and if the password meets complexity requirements (e.g., min 8 chars, 1 uppercase, 1 symbol).
Supabase Profile Duplication Check:
Before calling any Auth service, the Controller must query the profiles table. If a record with that email exists, the request is aborted with a 409 Conflict error to prevent redundant data.
Hard-Coded Role Assignment:
The logic must prevent a user from registering as both roles simultaneously with one ID. A conditional check if (role !== 'Rider' && role !== 'Driver') rejects the request. The role is then permanently locked to that UUID in the database.
Password Hashing:
Passwords must never reach the DB in plain text. Use bcrypt.hash() in a authUtils.ts file. This ensures that even if the database is compromised, user accounts remain secure.

1.2 Login & Session Logic

Hash Comparison:
The backend retrieves the stored hash and uses bcrypt.compare() to verify the incoming password attempt.
Dual Token Generation:
Logic issues an Access Token (JWT) for immediate API calls and a Refresh Token. This setup allows users to stay logged in without sending their password over the network repeatedly.
Authenticate Guard Middleware:
A custom middleware function that intercepts every protected request. It pulls the token from the Authorization: Bearer header, verifies it, and attaches req.user = decodedToken so every route knows exactly who is making the request.

1.3 Profile Logic & Media

Image Upload & Unique Filenaming:
When a user uploads a profile picture, the logic must generate a unique filename (e.g., user-id-timestamp.jpg). It then streams the file to Supabase Storage and saves the returned Public URL into the user's row in the PostgreSQL profiles table.

Phase 2: Driver Verification & Compliance (The Trust Layer)

2.1 Document Submission

Multipart Form Handling:
Using middleware like Multer, the backend accepts multiple files (License, Insurance, RC). These are routed to specific folders in Supabase Storage buckets.
Default Pending State:
The logic initializes every new driver with is_verified = false and account_status = 'PENDING_APPROVAL'.

2.2 Admin Approval Flow

Verification Toggle:
A protected PATCH /admin/verify-driver route only accessible to admin-role users. It toggles the is_verified boolean.
Logic Gate for Online Status:
Inside the Go Online route, the logic must check: if (!driver.is_verified) return Error("Account not verified"). This prevents unverified drivers from appearing on the map.

2.3 Vehicle Linking

Anti-Ghosting Logic:
The backend ensures a 1:1 link between a Driver ID and a Vehicle ID. It checks if the License Plate is already active on another account to prevent fraudulent "Ghost" cars.

Phase 3: Real-Time Movement & Geospatial (The Pulse)

3.1 Availability & Heartbeat

Availability Toggle:
Updates the is_available boolean and last_online timestamp. This is the primary switch for the matching engine.
Optimized Location Updates:
To save DB resources, the logic compares the new coordinates with the stored coordinates. It only updates the database if the distance moved is greater than 10 meters.

3.2 Nearby Search (PostGIS)

Geospatial Query:
Uses the ST_DWithin function in PostGIS to find drivers whose location point is within a specific radius of the Rider's pickup point.
Radius Expansion Logic:
If the initial query (e.g., 2km) returns zero drivers, the logic automatically triggers a second query with a 5km radius to ensure the rider finds a car.

Phase 4: Ride Lifecycle & State Machine (The Core)

4.1 Request & Ranking

Price Estimation:
A Service calculates: (Base_Fare) + (Distance_KM * Rate) + (Estimated_Minutes * Rate). This must be calculated server-side to prevent tampering.
Matching Algorithm:
The backend ranks found drivers by a combined score of Distance and User Rating.
Socket Notification:
Utilizing Socket.io, the backend emits a "NEW_RIDE_REQUEST" event specifically to the top 5 nearest drivers.

4.2 Acceptance (The Race Condition)

Atomic Database Transaction:
Crucial logic: Use BEGIN and COMMIT. The logic checks if status === 'SEARCHING'. If true, it updates to ACCEPTED. This ensures that even if 10 drivers click "Accept" at once, only one succeeds.

4.3 Trip Security & Completion

OTP Handshake:
Backend generates a 4-digit OTP. The Start Trip route requires this OTP. If inputOTP === dbOTP, the status moves to ONGOING.
Completion & Reset:
When status hits COMPLETED, the backend automatically sets the driver's is_available back to true and triggers the payment calculation.

Phase 5: Payments, Surge & Payouts (The Money)

5.1 Surge & Transactions

Surge Pricing Logic:
Logic checks the Request_Count / Available_Driver_Count ratio in a 1km zone. If demand is > 2.0x supply, the price is multiplied by a surge factor.
Split Logic:
The backend calculates the 80/20 split: Platform_Fee = Total * 0.20 and Driver_Pay = Total * 0.80.
Wallet Management:
Increments the driver's virtual wallet. Logic prevents "Payout" requests if the balance is below a minimum threshold.

Phase 6: Post-Trip, Safety & Support

6.1 Safety & Feedback

Rating Recalculation:
When a rating is saved, the logic triggers an average update: (Sum of all star rows) / (Count of total rides).
Cancellation Penalty:
If ride.status === 'ACCEPTED' and time_passed > 5 mins, the backend flags the rider's account for a "Cancellation Fee" on the next billing.
Emergency SOS:
A high-priority route that immediately logs the ride ID, driver location, and rider info into a security_alerts table for instant admin visibility.
Note for Implementation: Every "Logic" point above should be handled in a dedicated Controller or Service. Use PostgreSQL Transactions for any step involving money or status changes to maintain data integrity.
