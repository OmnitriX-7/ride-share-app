Frontend Engineering: The Absolute 100% Master Specification

Phase 1: Identity, Security & Auth Architecture

REACT + JWT + BROWSER-STORAGE

1.1 Advanced Registration & Role Logic

The "Wizard" State Logic
The registration page must operate as a state-driven wizard. If the user selects "Driver," the frontend must use Conditional Rendering to inject Vehicle fields (Plate, Model) into the DOM. Before the data is sent, the Client-side Validation Logic must run: 1. Check email regex. 2. Verify password length. 3. Ensure role-specific fields aren't empty. If invalid, the "Submit" button remains disabled via disabled={!isValid}.
1.2 The JWT Handshake & Persistence Logic
When the login API returns a 200 OK, the Session Logic executes: 1. The accessToken is saved in the global state (Redux/Context) for high-speed API authorization headers. 2. The refreshToken is persisted in localStorage or a Secure Cookie. 3. A Protected Route Wrapper is used to check the presence of this token. If a user tries to access /rider-dashboard without a token, the logic triggers navigate('/login') automatically.

Phase 2: Real-Time Movement & Geospatial UX

MAPBOX GL + GEOLOCATION API + SOCKETS

2.1 The Moving Car (Interpolation) Logic

Smoothing the Driver Movement
In the Rider app, cars should not "snap" from one coordinate to another. The Animation Logic works as follows: When a Socket event DRIVER_LOCATION_UPDATE arrives, the frontend stores the previous_lat_long and new_lat_long. Instead of updating the marker instantly, use requestAnimationFrame or a library like Framer Motion to slide the marker icon over a 3-second duration. This creates the visual effect of the car driving.
2.2 Geocoding & Address Logic
Riders rarely know their exact Latitude/Longitude. The Reverse Geocoding Logic must trigger when the map initializes. It sends the user's GPS point to the Google/Mapbox API and converts it into a string (e.g., "Park St, NY"). This string is then populated into the "Pickup Location" input field via setValue().

Phase 3: The Matchmaking State Machine

PLACES API + DEBOUNCING + STATE SYNC

3.1 Destination Search & Debounce Logic

API Cost & Speed Optimization
As the user types into the destination bar, the app uses Debounce Logic. The API call is delayed by 300ms. If the user types another letter before 300ms, the previous timer is cleared with clearTimeout(). This ensures we only call the Google Places API when the user pauses, saving thousands of unnecessary API credits.
3.2 The "Searching" UI Logic
When "Request Ride" is clicked, the UI enters a Polling/Socket Listener State. A circular radar animation starts. Simultaneously, a client-side timer is set for 60 seconds. If the Socket doesn't receive RIDE_ACCEPTED within this time, the Timeout Logic triggers a modal saying "No Drivers Found" and resets the ride state to IDLE.

Phase 4: Driver Operations & Active Trips

NOTIFICATION API + AUDIO + SOCKET.IO

4.1 The Incoming Request "Flash" Logic

Attention-Grabbing Logic
When a Driver is "Online," the frontend listens for a NEW_RIDE_NEARBY socket. The Overlay Logic renders a full-screen modal that covers all other components. The Audio Logic triggers new Audio('/alert.mp3').play() to alert the driver. A 15-second visual countdown starts. If "Accept" is clicked, the app sends a PUT request; if the countdown ends, the Auto-Reject Logic closes the modal.
4.2 OTP Verification Logic
Once at the pickup location, the Driver UI displays a 4-digit numeric keypad. The Input Validation Logic checks the length. Once the 4th digit is entered, it automatically calls POST /verify-otp. The trip status only changes to ONGOING once the backend returns a success message.

Phase 5: Financials, SOS & Safety Logic

STRIPE ELEMENTS + GEOLOCATION + SOS

5.1 The Emergency SOS Logic

The "Panic" Trigger
During an active trip, an SOS button is always visible. The Hold-to-Trigger Logic requires the user to press and hold for 3 seconds (to prevent accidents). The logic then captures the most precise high_accuracy GPS coordinate and sends an immediate POST /sos to the backend with the current ride_id.
5.2 Payment & Fare Breakdown UI
Once the ride hits COMPLETED, the Fare Summary Logic displays a breakdown: Base + Distance + Surge. If surge_multiplier > 1.0, the UI highlights the surge component in red. The app then waits for a Socket event PAYMENT_PROCESSED before allowing the user to navigate back to the home map.

Phase 6: Professional Polish (The Final 5%)

SKELETONS + ERROR BOUNDARIES + CACHING

6.1 UX Smoothness & Connectivity Logic

Skeleton Loading & Connectivity
1. When fetching Ride History, use Skeleton Screens (grey pulsing cards) instead of spinners to reduce perceived wait time. 2. Implement Offline Detection Logic: Use window.addEventListener('offline') to show a banner saying "Connection Lost. Retrying..." and disable the "Request Ride" button to prevent errors. 3. Deep Linking Logic: If the user opens a notification while the app is closed, the logic must parse the URL and navigate directly to /trip/:ride_id.
ARCHITECTURE WARNING: All of the above logic must be encapsulated in Custom Hooks (e.g., useRide(), useSocket(), useLocation()). Never put map logic inside the Auth component. Keep your concerns separated to avoid a "Spaghetti Code" mess.
